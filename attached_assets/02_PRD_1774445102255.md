## 2. INQUADRAMENTO DEL PROBLEMA (PROBLEM FRAMING)

**Problem Statement:** Oggi l'HR Manager che usa una piattaforma HR incontra un'interruzione di workflow quando cerca di tradurre una RAL in netto mensile stimato, il che causa rallentamenti nel processo di offerta e comunicazioni imprecise al candidato.

**Root Cause:** Il calcolo retributivo italiano è strutturalmente opaco. Coinvolge 6+ variabili interdipendenti (INPS, IRPEF progressiva, detrazioni a formula, addizionali locali, bonus cuneo, trattamento integrativo) che cambiano ogni anno. Nessuno strumento integrato nelle piattaforme HR risolve questo nel contesto d'uso.

**Costo dell'inazione:** Ogni offerta formulata senza stima netta è un rischio di disallineamento con il candidato. Tempo perso: 5-15 minuti per simulazione × N offerte/settimana.

---

## 3. OBIETTIVI E METRICHE DI SUCCESSO

**Outcome desiderato:** L'HR Manager ottiene una proiezione netta affidabile in < 10 secondi senza uscire dal contesto di lavoro.

**North Star Metric (prototipo):** Correttezza del calcolo — delta ≤ 2% rispetto a un calcolatore di riferimento (PMI.it o simulazione commercialista) sullo stesso caso standard.

**Key Results:**

| KR | Valore attuale | Target |
|----|---------------|--------|
| Tempo per ottenere una proiezione netta | 5-15 min (tool esterno) | < 10 secondi |
| Voci dettagliate visibili | 0 (la maggior parte dei tool dà solo il totale) | 8+ voci nel breakdown |
| Semplificazioni dichiarate | 0 (nessun tool le dichiara) | Tutte elencate nella UI |

**Contro-metrica (guardrail):** Il tool NON deve comunicare precisione che non ha. Se un utente lo usa come sostituto del cedolino paga, è un fallimento di design.

## 5. REQUISITI FUNZIONALI E USER STORIES

### US-01: Calcolo base
*Come HR Manager, voglio inserire una RAL e ottenere il netto annuale e mensile, affinché possa stimare rapidamente il netto di un'offerta.*

**Acceptance Criteria:**
- [ ] L'utente inserisce un importo numerico (RAL) e clicca "Calcola"
- [ ] Il sistema mostra: netto annuale, netto mensile (÷13), aliquota effettiva
- [ ] Il calcolo si completa in < 1 secondo
- [ ] L'input accetta valori tra €1 e €500.000
- [ ] L'input formatta automaticamente i separatori delle migliaia

### US-02: Breakdown dettagliato
*Come HR Manager, voglio vedere il dettaglio di ogni voce trattenuta, affinché possa capire e spiegare perché il netto è quello.*

**Acceptance Criteria:**
- [ ] Il breakdown mostra: contributi INPS, imponibile IRPEF, IRPEF lorda, detrazioni lavoro dipendente, ulteriore detrazione cuneo (se applicabile), IRPEF netta, addizionale regionale, addizionale comunale, bonus cuneo (se applicabile), trattamento integrativo (se applicabile)
- [ ] Ogni voce mostra l'importo annuale e la percentuale sulla RAL
- [ ] Le voci positive (detrazioni, bonus) sono visivamente distinte dalle trattenute
- [ ] Il netto finale è evidenziato

### US-03: Visualizzazione distribuzione
*Come HR Manager, voglio una vista sintetica di come la RAL si distribuisce tra netto e trattenute, affinché possa avere un colpo d'occhio immediato.*

**Acceptance Criteria:**
- [ ] Una barra orizzontale mostra la proporzione: netto / INPS / IRPEF / addizionali
- [ ] I segmenti sono colorati e hanno una legenda con importi e percentuali
- [ ] L'hover su un segmento mostra il dettaglio

### US-04: Note e semplificazioni
*Come utente, voglio sapere quali assunzioni il calcolatore sta facendo, affinché non confonda la stima con un cedolino ufficiale.*

**Acceptance Criteria:**
- [ ] Una sezione "Note e semplificazioni" è sempre visibile nei risultati
- [ ] Elenca: anno fiscale, tipo contratto, residenza, mensilità, assenza carichi di famiglia
- [ ] Elenca le semplificazioni tecniche rilevanti (es. trattamento integrativo parziale)

**Priorità:**

| User Story | Priorità | Motivazione |
|-----------|----------|-------------|
| US-01 | P0 (Must have) | Senza questo il tool non esiste |
| US-02 | P0 (Must have) | Il breakdown è il differenziatore — senza, è uguale a PMI.it |
| US-03 | P1 (Should have) | Aggiunge valore visivo ma non blocca il job |
| US-04 | P0 (Must have) | Senza disclaimer il tool è irresponsabile |

## 6. CASI LIMITE E GESTIONE ERRORI

### Input

| Caso | Comportamento atteso |
|------|---------------------|
| RAL = 0 o vuota | Messaggio: "Inserisci una RAL valida" |
| RAL < €8.500 (no-tax area) | Calcolo corretto: IRPEF = 0 dopo detrazioni. Il tool deve funzionare, non bloccare |
| RAL molto alta (> €200.000) | Calcolo corretto con tutti gli scaglioni. Nota: detrazioni lavoro dipendente = 0 sopra €50K |
| RAL con decimali | Arrotondamento all'euro intero |
| Input non numerico | Filtro: accetta solo cifre, formatta con separatori |

### Calcolo

| Caso | Comportamento atteso |
|------|---------------------|
| IRPEF netta negativa (detrazioni > imposta lorda) | IRPEF netta = 0 (le detrazioni non generano credito nel calcolo mensile) |
| RAL nella fascia bonus cuneo (≤ €20K) | Il bonus appare come voce positiva nel breakdown |
| RAL nella fascia ulteriore detrazione (€20K-€40K) | La detrazione appare nel breakdown |
| RAL nella fascia esenzione comunale Milano (≤ €23K imponibile) | Addizionale comunale = 0, nota "esente" |
| Contributo aggiuntivo INPS 1% (RAL > ~€60K) | Voce separata nel breakdown, non sommata silenziosamente |

### UI

| Caso | Comportamento atteso |
|------|---------------------|
| Ricalcolo con nuova RAL | I risultati precedenti vengono sostituiti, non accumulati |
| Schermo mobile (< 600px) | Layout responsive: summary cards su 2 colonne, breakdown full-width |
| Invio con Enter | Equivale a click su "Calcola" |

## 7. SPECIFICHE TECNICHE E VINCOLI

**Stack:**

| Componente | Scelta | Motivazione |
|-----------|--------|-------------|
| Runtime | Client-side (browser) | Zero backend, zero dati sensibili, massima portabilità |
| Linguaggio | HTML + CSS + JavaScript vanilla | Nessuna dipendenza, file singolo, ispezionabile |
| Hosting | File statico (apribile da browser) | Il prototipo deve essere testabile con un doppio click |

**Vincoli:**

- Nessun dato personale raccolto → GDPR non applicabile (nessun input persistito)
- Nessuna API esterna → funziona offline
- File singolo HTML → nessun build step, nessun framework
- Le costanti fiscali (aliquote, soglie) devono essere raggruppate in un oggetto configurazione in cima al codice → aggiornamento annuale semplice

**Performance:** Il calcolo è pura aritmetica. Target: < 50ms su qualsiasi device.

## 8. UX, DESIGN E ACCESSIBILITÀ

**Principi:**

- Mobile-first responsive
- Riduzione del carico cognitivo: un input, un bottone, risultato immediato
- Il breakdown è leggibile dall'alto al basso come un ragionamento: da RAL → a netto
- Nessuna animazione che rallenti la percezione del risultato

**User Flow:**

1. L'utente vede il campo RAL → inserisce un numero
2. Clicca "Calcola" (o preme Enter)
3. Sotto l'input appaiono: summary cards, barra distribuzione, breakdown dettagliato, note
4. Per ricalcolare: modifica il numero e clicca di nuovo

**Accessibilità:**

- Contrasto testo/sfondo ≥ 4.5:1 (WCAG AA)
- Input con label esplicita
- Navigazione da tastiera funzionante (tab → input, enter → calcola)
- Font size minimo 13px per il body

Specifiche visive nel documento 04_DESIGN_SYSTEM.md.

## 9. PIANO DI ANALYTICS E TRACCIAMENTO

*Per un prototipo standalone non c'è analytics reale. Questa sezione documenta cosa tracceremmo in produzione.*

| Evento | Trigger | Proprietà |
|--------|---------|-----------|
| `ral_calculator_opened` | Apertura pagina/modale | `source` (da dove viene l'utente) |
| `ral_calculated` | Click su "Calcola" | `ral_input`, `netto_result`, `aliquota_effettiva` |
| `breakdown_viewed` | Scroll fino al breakdown (viewport) | `ral_input` |
| `recalculation` | Secondo click "Calcola" nella stessa sessione | `ral_previous`, `ral_new` |

**Ipotesi da validare con i dati:**

- Qual è la RAL più inserita? (indica il segmento che usa di più il tool)
- Quanti utenti ricalcolano? (indica che stanno facendo simulazioni, non un check singolo)
- Quanti scrollano fino alle note? (indica se il disclaimer è letto)
